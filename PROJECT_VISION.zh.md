# Claude Trading Skills 项目愿景

版本：0.1
最后更新：2026-05-17

English version: [PROJECT_VISION.md](PROJECT_VISION.md) ・ 日本語版: [PROJECT_VISION.ja.md](PROJECT_VISION.ja.md)

Claude Trading Skills 是一套基于 Claude Skills、面向时间受限个人投资者的交易流程操作系统（OS）。

它帮助投资者以长期投资为根基，仅在市场条件允许时把握有纪律的波段交易机会，并通过风险管理、记录与复盘持续改进。

## 1. 信条（Mantra）

**Empower solo traders, growing together.（赋能独立交易者，共同成长。）**

本项目的存在，是为了帮助个人交易者走出孤立的随性决策，用 Claude Skills 建立可复用的流程、明确的风险管理、记录、复盘与持续改进的循环。

这里的 **solo（独立）** 并不意味着孤立。本项目为自己做决策、自己承担风险的交易者而设计，同时让工作流、复盘与改进实践通过共享实践不断成长。

## 2. 重要声明与免责

本仓库提供的 Claude Skills 及相关材料仅用于教育、研究与流程改进目的。它不提供财务建议、投资顾问服务、交易信号、券商执行、税务建议或法律建议。

投资与交易有风险，可能损失本金。历史数据、回测、筛选结果、示例报告及 AI 生成的分析均不保证未来结果。所有交易决策、仓位测算、风险管理、税务/合规以及券商使用决策，均完全由用户自行负责。

本项目以 MIT 许可证提供。软件与材料按许可证所述，以 **“原样（AS IS, WITHOUT WARRANTY）”** 提供，不附带任何担保。

## 3. 起源与开源的原因

本项目的起点，是作者希望用 AI 改进自己的交易流程。

对个人交易者而言，尤其是那些一边投资交易、一边还要应对工作、家庭或其他生活责任的人，时间、信息过载、情绪控制与风险管理都是实实在在的约束。Claude Trading Skills 的创建，正是为了在这些约束下，更容易地维持一套用于市场复盘、候选生成、交易计划、风险管理、记录与复盘的流程。

本项目首先是作者每日、每周自用的实用工具。它的存在，是为了改进作者自己的交易流程、构建更具韧性的决策流程，并持续从记录与复盘中学习。与此同时，它之所以开源，是因为同样的工作流可能对面临类似约束的其他人有用。

立场是 **first for self, open for others（先为自己，再开放给他人）**。第一位用户是作者。即使外部反馈有限，本项目也应作为作者真正使用的工具持续改进。在此基础上，本项目希望让工作流、复盘与改进实践，作为个人交易者的共享实用知识不断积累。

Claude Trading Skills 并不提供一套现成的“制胜公式”。作者自己也是仍在学习的交易者。本仓库不是发放“答案”的地方，而是分享实用工作流的地方——帮助个人交易者强化判断、管理风险、从记录中学习、并逐步改进。

## 4. 核心论点

本项目的目标，不是比所有人都更会预测市场。

目标是帮助个人投资者与交易者，做出更结构化、更具风险意识、可复盘、可改进的决策。

本项目的核心流程不是：

```text
提问 -> 信号 -> 交易
```

核心循环是：

```text
计划 -> 交易 -> 记录 -> 复盘 -> 改进
```

Claude Trading Skills 不是信号引擎。它的目标是成为一套 **决策流程操作系统（decision-process OS）**，帮助个人交易者用 AI 构建更具韧性的决策流程，并持续管理判断、风险、记录与改进。

## 5. 目的

Claude Trading Skills 是一个 Claude Skills 仓库，聚焦于美股、ETF、股息投资、波段交易、宏观分析与策略研究，并在适当之处纳入期权与事件驱动策略。

这些技能支持如下决策：

- 理解市场状况
- 筛选交易或投资候选
- 构建交易计划
- 计算仓位规模
- 复盘组合风险
- 记录交易论点
- 复盘结果并识别改进点
- 测试新的策略想法

本项目不只是一组有用的分析工具的集合。它的目标是成为面向个人交易者的 **决策支持系统**。

特别地，它希望帮助这样的个人投资者：以长期投资为根基，希望在市场条件允许时高效利用有限时间做波段交易，既不承担过度风险，又通过记录与复盘不断改进。

## 6. 本项目是什么 / 不是什么

| 本项目是 | 本项目不是 |
| --- | --- |
| 决策支持系统 | 财务建议或投资顾问服务 |
| 交易工作流工具箱 | 买卖信号服务 |
| 风险管理与复盘框架 | 盈利保证系统 |
| Claude Skills 仓库 | 券商执行平台 |
| 支持交易者学习循环的系统 | 全自动交易机器人 |

本项目不替代交易者的判断。它帮助把交易者的决策流程变得显式、可复用、可复盘、可改进。

最终决策与风险责任始终归用户所有。详细免责声明集中在“重要声明与免责”一节，随着项目成长，这一立场不应改变。

## 7. 目标用户

主要受众是 **时间受限的个人投资者与波段交易者**。

更具体地说，本项目为这样的人设计：以长期投资与股息投资作为财富积累的根基，同时在市场条件有利时，通过中短期波段交易寻求额外收益。他们有工作、家庭或生活上的约束，因此需要让日常决策更高效，并把风险管理与记录系统化。

### 主要用户

最优先服务的首批用户是：

| 角色 | 主要目标 | 所需路径 |
| --- | --- | --- |
| 兼职波段交易者 | 每日市场复盘、候选生成、交易计划 | `market-regime-daily` / `swing-opportunity-daily` |
| 想要更强风险控制的成长型投资者 | 区分进攻型市场与防御型市场 | `market-regime` / `exposure` 工作流 |
| 股息与长期投资者 | 寻找股息股、复盘持仓、检查组合结构 | `core-portfolio` / `dividend-income` |

### 次要 / 进阶用户

本项目也能支持：

| 角色 | 主要目标 | 所需路径 |
| --- | --- | --- |
| 事件 / 财报交易者 | 在财报、新闻或经济事件后寻找机会 | `advanced-satellite` / `earnings-event` |
| 做空策略交易者 | 在 risk-off 环境中监控弱势股或过度拉伸的股票 | `advanced-satellite` / `risk-off-short` |
| 策略研究者 / 开发者 | 检验假设并改进策略 | `strategy-research` |
| 进阶用户 | 扩展自定义工作流、YAML manifest 与 CLI 工具 | manifests / scripts / API 矩阵 |

本项目并非主要为以下人群设计：

- 期望全自动交易的用户
- 想要保证盈利或外包交易信号的用户
- 不愿管理风险或记录的用户
- 只专注于短线刷单（scalping）的用户

## 8. Core + Satellite 运营理念

本项目的主要用户以 **Core + Satellite（核心 + 卫星）** 结构运营：

- **Core（核心）**：长期投资、股息股、ETF、组合管理
- **Satellite（卫星）**：波段交易、主题、突破、财报后动量
- **Advanced Satellite（进阶卫星）**：做空策略、事件策略，以及在适当之处的期权策略
- **共享层（Shared Layer）**：市场环境、风险管理、仓位测算、记录、复盘

关键在于不要把 Core 与 Satellite 活动的目的、时间框架与风险特征混在一起。

本项目为这样的个人投资者提供一套运营流程：以长期财富积累为根基，仅在市场条件允许时，寻求有纪律的中短期交易机会。

Advanced Satellite 包含风险更高、假设更复杂或执行约束更多的领域。做空策略、事件策略与期权策略，只应在具备足够经验、设有明确止损上限、有人工复核、理解 API 与券商约束、并有预先验证过的工作流的前提下使用。一个实现得很重的技能，未必属于主要用户的首选推荐路径。

## 9. 当前状态

本仓库已包含许多技能。大致涵盖：

- **市场分析**：市场环境、宽度、板块、宏观、新闻、泡沫风险
- **筛选**：CANSLIM、VCP、股息股、财报后动量、主题、机构资金流
- **交易计划**：突破、抛物线做空（Parabolic Short）、仓位测算、敞口管理
- **组合与记录**：组合管理、论点跟踪、交易记录、事后复盘
- **策略研究**：回测、策略想法生成、edge 研究流水线
- **质量与元技能**：技能设计、评审、集成测试、改进循环

各个技能已颇具规模。但从用户视角看，仍存在若干问题：

- 难以知道该用哪个技能
- 难以看清多个技能应如何排序衔接
- 交易前后的完整工作流尚未清晰打包
- 对技术不熟悉的用户而言，GitHub 与 `.skill` 文件可能令人望而却步
- 技能输出尚未完全接入持续学习循环

下一阶段的重点，不在于增加更多技能，而在于 **把技能打包成可用的形态**、**引导用户走向正确路径**，以及 **将其转化为运营工作流**。

## 10. 战略方向：从技能集合到 Trading OS

项目的下一步方向可概括为：

> 从技能集合，走向面向独立交易者的 Trading OS。

所谓 Trading OS，是指一个工作流层，把市场背景、风险预算、策略选择、候选生成、交易计划、人工执行关卡、记录、复盘与改进连接起来，服务于每日与每周的投资决策。

这套 Trading OS 并非为了让个人交易者轻松取胜，而是为了帮助他们留在牌桌上、管理风险、从记录中学习，并在时间有限的情况下逐步改进决策流程。

理想的体验如下：

1. 用户用自然语言描述他想做什么样的交易或投资
2. 系统理解其目标、经验、时间预算、风险承受度与 API 环境
3. 它推荐合适的技能集与工作流
4. 用户可按顺序复盘市场状况、风险、候选与交易计划
5. 交易后，用户记录论点与结果，并把心得反馈进下一次迭代

如果这条流程跑得通，即使是对 GitHub 或工具配置不熟悉的用户，也能一步步获得本项目的价值。

## 11. 项目架构

本项目应按以下层次组织：

```text
1. 单个技能（Individual Skills）
    |
2. 技能清单 / API 矩阵（Skill Inventory / API Matrix）
    |
3. 技能集（Skillsets）
    |
4. 工作流（Workflows）
    |
5. Trading Skills Navigator（导航器）
    |
6. 用户入口（User Entry Points）
    |
7. 日志 / 事后复盘 / 学习循环（Journal / Postmortem / Learning Loop）
```

每一层都有明确职责：

| 层 | 职责 |
| --- | --- |
| 技能（Skills） | 用于分析、计算、计划与记录的小型专精能力 |
| 技能清单 / API 矩阵 | 关于用例、所需 API、难度、输入/输出的单一事实来源 |
| 技能集（Skillsets） | 按目标划分的捆绑，定义哪些技能该组合在一起 |
| 工作流（Workflows） | 运营序列、判断关卡与产物交接 |
| 导航器（Navigator） | 根据用户目标推荐技能集与工作流的交互式向导 |
| 用户入口 | 文档、起步提示词、CLI、未来的 Web UI 等开始使用项目的方式 |
| 学习循环 | 让技能与交易者共同成长的记录、复盘与改进机制 |

这一结构减少了技能源文件、文档、推荐逻辑与工作流之间的漂移。导航器提供交互式推荐，用户入口则提供静态文档、起步提示词、CLI 工具与未来的 Web UI 路径。

> **状态：** PR #85 在 `workflows/` 下提供了 5 个 Core + Satellite manifest（`core-portfolio-weekly`、`market-regime-daily`、`swing-opportunity-daily`、`trade-memory-loop`、`monthly-performance-review`），由 `--strict-workflows` 校验。进阶工作流（`risk-off-short-daily`、`earnings-weekly`、`strategy-research-pipeline`）仍为后续工作。

仅有技能集不足以支撑真实运营。交易需要序列、判断关卡与产物交接。

一条典型工作流：

1. **市场背景（Market Context）**：复盘市场状况
2. **风险预算（Risk Budget）**：决定承担多少风险
3. **策略选择（Strategy Selection）**：选择今天允许哪种策略
4. **候选生成（Candidate Generation）**：寻找候选
5. **交易计划（Trade Planning）**：定义入场 / 止损 / 目标 / 仓位
6. **人工执行关卡（Manual Execution Gate）**：任何真实下单前先复核
7. **监控（Monitoring）**：监控触发与失效
8. **日志 / 事后复盘（Journal / Postmortem）**：记录与复盘

初始候选：

- `core-portfolio-weekly`
- `market-regime-daily`
- `swing-opportunity-daily`
- `trade-memory-loop`
- `monthly-performance-review`

进阶工作流候选：

- `risk-off-short-daily`
- `earnings-weekly`
- `macro-morning-brief`
- `strategy-research-pipeline`

完成标准：

- 至少 3 条运营工作流以 YAML 定义
- 每条工作流都有输入、输出、判断关卡、所用技能与人工复核项
- 存在兼职交易者可在 15-60 分钟内运行的每日 / 每周路径
- 每条交易工作流都连接到一条日志条目或事后复盘
- 至少 1 条工作流能用示例数据端到端讲解

## 14. 设计原则

未来开发应遵循以下原则：

1. **流程优先于预测**
   - 优先打造可复用的决策流程，而非预测本身。

2. **风险优先**
   - 在候选或信号之前，先复盘市场状况与风险预算。

3. **人的判断居于核心**
   - 本项目支持人的判断，而非自动化交易决策。

4. **推荐必须可解释**
   - 系统应解释为何建议某个技能、工作流或风险设置。

5. **组合小型技能**
   - 优先把小而专注的技能组合起来，而非打造庞大的万能技能。

6. **对新手可达，对进阶用户可扩展**
   - 为非 GitHub 用户提供简单入口，同时为深度自定义保留空间。

7. **记录并改进**
   - 一笔交易不在执行时结束；它在被记录、复盘并用于改进时才结束。

8. **维护单一事实来源**
   - 技能元数据应尽量集中在尽可能少的正本之处。目录、API 矩阵、导航器推荐与工作流 manifest 应尽可能由共享元数据生成或对其校验。

## 17. 长期愿景

长期而言，本项目应成为这样一个地方：

- 个人交易者能找到适合自己的策略与工作流
- 市场环境能指导风险敞口
- 交易计划在执行前是显式的
- 交易后记录结果并用于学习
- 技能与工作流通过实践不断改进
- 新手有平易近人的入口，进阶用户有扩展的空间

最终，用户应能分享工作流配方、事后复盘模板、策略研究笔记与技能改进提案，让个人交易者的实用知识改进整个项目。

本项目的本质，不是把“答案”交给交易者。

而是营造一个环境，让交易者能强化判断、管理风险、从记录中学习，并持续成长。

**Empower solo traders, growing together.（赋能独立交易者，共同成长。）**
