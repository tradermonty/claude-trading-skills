# 見積依頼書（RFQ）作成チェックリスト

このチェックリストは、ベンダーに提出する見積依頼書を作成する際に、必要な情報が網羅されているかを確認するためのものです。

## 1. プロジェクト概要

### 1.1 基本情報
- [ ] プロジェクト名が明確に定義されている
- [ ] プロジェクトの背景・経緯が説明されている
- [ ] プロジェクトの目的・ゴールが明確である
- [ ] ビジネス上の課題・解決したい問題が記載されている
- [ ] ステークホルダーが特定されている

### 1.2 プロジェクトスコープ
- [ ] プロジェクトの範囲（スコープ）が明確である
- [ ] 含まれる機能・サービスが列挙されている
- [ ] スコープ外（除外事項）が明示されている
- [ ] 制約事項が記載されている
- [ ] 前提条件が明確である

### 1.3 成果物
- [ ] 期待される成果物がリストアップされている
- [ ] 各成果物の受入基準が定義されている
- [ ] ドキュメント類の要件が明確である
- [ ] ソースコードの納品形式が指定されている
- [ ] 知的財産権の帰属が明記されている

## 2. 機能要件

### 2.1 ユーザー要件
- [ ] 対象ユーザー・ユーザー層が特定されている
- [ ] ユーザーストーリー/ユースケースが記載されている
- [ ] ユーザーの業務フロー・プロセスが説明されている
- [ ] ユーザーインターフェース要件が定義されている
- [ ] アクセシビリティ要件が含まれている

### 2.2 システム機能
- [ ] 主要機能が優先順位付きでリストアップされている
- [ ] 各機能の詳細仕様が記載されている
- [ ] 機能間の関連・依存関係が明確である
- [ ] データ入出力の要件が定義されている
- [ ] ビジネスロジック・計算ルールが明確である

### 2.3 統合要件
- [ ] 既存システムとの連携要件が定義されている
- [ ] 外部サービス・APIとの統合が明記されている
- [ ] データ移行の要件が含まれている（該当する場合）
- [ ] インターフェース仕様が提供されている
- [ ] 認証・認可の統合方式が明確である

## 3. 非機能要件

### 3.1 パフォーマンス要件
- [ ] 応答時間の要件が定義されている
- [ ] スループット（処理能力）が明記されている
- [ ] 同時接続ユーザー数が指定されている
- [ ] データ量の見込みが記載されている
- [ ] ピーク時の負荷要件が含まれている

### 3.2 可用性・信頼性
- [ ] システム稼働時間・ダウンタイム許容度が定義されている
- [ ] 障害復旧時間目標（RTO）が明記されている
- [ ] 目標復旧時点（RPO）が定義されている
- [ ] バックアップ要件が含まれている
- [ ] 災害対策（DR）要件が明確である

### 3.3 セキュリティ要件
- [ ] 認証方式が指定されている
- [ ] 認可・アクセス制御の要件が定義されている
- [ ] データ暗号化の要件が明記されている
- [ ] 通信のセキュリティ要件が含まれている
- [ ] 監査ログ・トレーサビリティ要件が定義されている
- [ ] コンプライアンス要件が明確である（GDPR、個人情報保護法など）
- [ ] 脆弱性診断・ペネトレーションテストの要否が記載されている

### 3.4 拡張性・保守性
- [ ] 将来的な拡張計画が記載されている
- [ ] スケーラビリティ要件が定義されている
- [ ] 保守性・メンテナンス性の要件が含まれている
- [ ] モジュール性・疎結合性の期待が明記されている
- [ ] ドキュメンテーション要件が定義されている

### 3.5 運用要件
- [ ] 監視・モニタリング要件が定義されている
- [ ] ログ管理要件が明記されている
- [ ] デプロイメント方式が指定されている
- [ ] 運用サポート体制の要件が含まれている

### 3.6 ユーザビリティ
- [ ] ユーザーエクスペリエンス（UX）の期待が明確である
- [ ] レスポンシブデザインの要否が記載されている
- [ ] 対応ブラウザ・デバイスが指定されている
- [ ] 多言語対応の要否が明記されている
- [ ] アクセシビリティ標準（WCAG等）への準拠要件が含まれている

## 使用方法

このチェックリストを使用して、見積依頼書を作成する際に以下のプロセスを実施してください：

1. **初期レビュー**: 顧客から受け取った要望をこのチェックリストと照合し、不足している情報を特定する
2. **ヒアリング**: 不足している情報について顧客にヒアリングを実施する
3. **文書化**: 収集した情報をもとに見積依頼書を作成する
4. **最終確認**: 完成した見積依頼書をこのチェックリストで再度確認し、すべての必須項目が含まれていることを検証する
5. **レビュー**: 関係者（プロジェクトマネージャー、技術リード、法務など）による最終レビューを実施する

## チェックリスト活用のベストプラクティス

### プロジェクトタイプ別の重点項目

**Webアプリケーション開発**:
- ユーザビリティ要件 (Section 3.6)
- パフォーマンス要件 (Section 3.1)
- セキュリティ要件 (Section 3.3)
- レスポンシブデザイン要件

**モバイルアプリ開発**:
- 対応OS・バージョン (Section 4.1)
- デバイス依存機能（GPS、カメラなど）
- オフライン動作要件
- アプリストア公開要件

**基幹システム開発**:
- 可用性・信頼性要件 (Section 3.2)
- データ移行要件 (Section 2.3)
- 既存システムとの統合 (Section 2.3)
- 運用要件 (Section 3.5)

**データ分析・AI/ML プロジェクト**:
- データ品質要件
- モデル精度要件
- 学習データの要件
- 説明可能性・透明性要件

### 優先順位付け

すべての項目をチェックすることが理想ですが、時間・リソースの制約がある場合は以下の優先順位で進めてください：

**必須 (Must Have)**:
- プロジェクト概要 (Section 1)
- 機能要件 (Section 2)
- スケジュール・予算 (Section 5.1, 5.2)
- 見積書の形式 (Section 7.1)
- 提出手続き (Section 7.5)

**重要 (Should Have)**:
- 非機能要件の主要項目 (Section 3.1-3.3)
- 技術要件 (Section 4)
- 品質管理 (Section 5.4)
- 契約要件 (Section 6)

**推奨 (Nice to Have)**:
- 非機能要件の詳細 (Section 3.4-3.6)
- 提案要件 (Section 7.3)
- 評価基準 (Section 7.4)
- 参考資料 (Section 8)
