# 工数見積基準ガイド

## Overview

このガイドは、システム開発における各タスクの標準工数、役割別生産性、およびプロジェクトタイプ別の工数見積基準を提供します。

---

## 役割別生産性指標

### コーディング生産性（LOC: Lines of Code）

| 役割 | 生産性（LOC/人日） | コード品質 | 適用範囲 |
|------|------------------|-----------|---------|
| **シニアエンジニア** | 150〜250 | 高品質、バグ少 | 複雑なロジック、アーキテクチャ |
| **ミドルエンジニア** | 100〜180 | 標準品質 | 通常のビジネスロジック |
| **ジュニアエンジニア** | 50〜100 | レビュー必要 | 単純なCRUD、UI実装 |

**注意**: LOCは言語により異なる
- Java/C#: 基準値
- Python/Ruby: 基準値 × 0.6〜0.7（簡潔な記述）
- JavaScript/TypeScript: 基準値 × 0.8〜0.9

---

### 機能ポイント（FP）生産性

| 役割 | 生産性（FP/人月） | 経験年数 | 備考 |
|------|-----------------|---------|------|
| **シニア** | 8〜12 FP/人月 | 5年以上 | 複雑度高い機能も対応 |
| **ミドル** | 5〜8 FP/人月 | 3〜5年 | 標準的な機能 |
| **ジュニア** | 3〜5 FP/人月 | 1〜3年 | 単純な機能 |

**換算**:
- 1 FP = 約50〜150 LOC（言語・複雑度による）
- 1人月 = 20人日（稼働日ベース）

## 調整係数の適用

### 複合調整例

**ベース工数**: 1,000人日
**プロジェクト特性**:
- 複雑度: 高（係数 1.3）
- チーム習熟度: 初級（係数 1.3）
- 技術リスク: 中（係数 1.2）

**調整後工数**:
```
1,000人日 × 1.3 × 1.3 × 1.2 = 2,028人日
```

ただし、複数係数を乗算すると過大になる場合があるため、実務では：
```
加重平均アプローチ:
1,000人日 × (1 + (0.3 + 0.3 + 0.2) / 3) = 1,267人日
```

または、最大リスク要因のみ適用：
```
1,000人日 × 1.3 = 1,300人日
```

## 工数見積の検証

### 妥当性チェック

#### 1. 生産性チェック

**FP/人月での検証**:
```
実装フェーズ工数: 400人日 = 20人月
機能ポイント: 150 FP

生産性 = 150 FP / 20人月 = 7.5 FP/人月
→ ミドルエンジニア中心（5-8 FP/人月）で妥当
```

#### 2. 類似プロジェクトとの比較

```
類似プロジェクトA: 画面40、工数900人日 → 22.5人日/画面
現在プロジェクト: 画面50、工数1,200人日 → 24人日/画面
差異: +7% → 許容範囲内
```

#### 3. フェーズ比率チェック

```
要件:設計:実装:テスト = 12:18:45:22
→ Webアプリの標準比率と一致
```

## まとめ

**工数見積のポイント**:
1. **タスク分解**: WBSで詳細に分解
2. **標準工数適用**: 役割とタスクに応じた標準工数
3. **調整係数適用**: プロジェクト特性に応じた調整
4. **検証**: 生産性指標、類似PJ、フェーズ比率で検証
5. **コンティンジェンシー**: 10-25%のバッファ確保

**定期的な見直し**:
- プロジェクト実績データを蓄積
- 標準工数を自社実績で更新
- 調整係数を自組織に合わせて調整
