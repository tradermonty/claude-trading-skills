# システム開発見積もり手法ガイド

## Overview

このガイドは、システム開発プロジェクトにおける見積もり手法と、精度の高い見積もりを作成するためのベストプラクティスを提供します。

## 見積もり手法の種類

### 1. 類推法（Analogous Estimation）

**概要**: 過去の類似プロジェクトのデータを基に見積もる手法

**適用場面**:
- プロジェクト初期段階（要件が不明確）
- 過去に類似プロジェクトの実績がある
- 迅速な概算が必要

**手順**:
1. 過去の類似プロジェクトを特定
2. 類似度を評価（規模、複雑度、技術スタック）
3. 差異を考慮して調整（調整係数: 0.8〜1.5）
4. 概算コストを算出

**精度**: ±25〜50%

**例**:
```
過去プロジェクト: ECサイト構築 - 1,200万円、6ヶ月
新規プロジェクト: 類似ECサイト、機能30%増
調整係数: 1.3
見積もり: 1,200万円 × 1.3 = 1,560万円
```

---

### 2. パラメトリック法（Parametric Estimation）

**概要**: 統計的な関係性（単価×数量）を用いて見積もる手法

**適用場面**:
- 過去データが豊富にある
- プロジェクト規模が測定可能
- 中程度の精度が必要

**主要パラメータ**:
- **機能ポイント（FP: Function Point）**: 機能の複雑度に基づく測定単位
- **画面数**: UI開発の規模指標
- **API数**: バックエンド開発の規模指標
- **DB テーブル数**: データ設計の規模指標

**手順**:
1. プロジェクト規模を測定（FP、画面数等）
2. 単価を設定（過去実績から）
3. 工数 = 規模 × 単価
4. コスト = 工数 × 人件費単価

**精度**: ±15〜30%

**例**:
```
規模: 画面30本、API 50本、DBテーブル25本
単価:
  - 画面: 8人日/画面
  - API: 3人日/API
  - DBテーブル: 2人日/テーブル

工数計算:
  - 画面: 30 × 8 = 240人日
  - API: 50 × 3 = 150人日
  - DB: 25 × 2 = 50人日
  - 合計: 440人日

コスト: 440人日 × 8万円/人日 = 3,520万円
```

---

### 3. ボトムアップ法（Bottom-Up Estimation）

**概要**: タスクを細分化し、個々のタスクを見積もって積み上げる手法

**適用場面**:
- 要件が明確
- 詳細な見積もりが必要
- 高精度が求められる

**手順**:
1. WBS（Work Breakdown Structure）を作成
2. 各タスクの工数を見積もり
3. タスクを積み上げて合計工数を算出
4. コンティンジェンシー（10-20%）を追加

**精度**: ±5〜15%

**WBS例**:
```
プロジェクト全体
├── 要件定義 (120人日)
│   ├── 機能要件定義 (60人日)
│   ├── 非機能要件定義 (40人日)
│   └── 要件レビュー (20人日)
├── 設計 (200人日)
│   ├── アーキテクチャ設計 (60人日)
│   ├── DB設計 (50人日)
│   ├── UI設計 (60人日)
│   └── 詳細設計 (30人日)
├── 実装 (400人日)
│   ├── バックエンド開発 (200人日)
│   ├── フロントエンド開発 (150人日)
│   └── API開発 (50人日)
├── テスト (250人日)
│   ├── 単体テスト (80人日)
│   ├── 統合テスト (100人日)
│   └── 受入テスト (70人日)
└── デプロイ・運用準備 (80人日)
    ├── 環境構築 (30人日)
    ├── データ移行 (30人日)
    └── 運用手順書作成 (20人日)

合計: 1,050人日
コンティンジェンシー (15%): 158人日
最終工数: 1,208人日
```

---

### 4. 三点見積もり（Three-Point Estimation / PERT）

**概要**: 楽観値、最頻値、悲観値の3つの見積もりを組み合わせる手法

**適用場面**:
- 不確実性が高い
- リスクを定量化したい
- 確率的な見積もりが必要

**手順**:
1. 楽観値（O: Optimistic）を見積もる（最良のケース）
2. 最頻値（M: Most Likely）を見積もる（通常のケース）
3. 悲観値（P: Pessimistic）を見積もる（最悪のケース）
4. 期待値を計算: E = (O + 4M + P) / 6
5. 標準偏差を計算: σ = (P - O) / 6

**精度**: ±10〜20%（信頼区間68%: E±σ、95%: E±2σ）

**例**:
```
タスク: API開発
- 楽観値 (O): 100人日（順調に進んだ場合）
- 最頻値 (M): 150人日（通常のケース）
- 悲観値 (P): 250人日（問題が多発した場合）

期待値 (E) = (100 + 4×150 + 250) / 6 = 158人日
標準偏差 (σ) = (250 - 100) / 6 = 25人日

結果:
- 68%確率: 158人日 ± 25人日 (133〜183人日)
- 95%確率: 158人日 ± 50人日 (108〜208人日)
```

## 見積もりベストプラクティス

### 1. 過去実績データの活用

**データ収集項目**:
- プロジェクト名、概要
- 規模（画面数、API数、テーブル数、FP）
- 実績工数（計画工数と実績工数の差異）
- 技術スタック
- チーム構成
- 主要課題と対応

**データベース例**:
```markdown
| プロジェクト | 規模 | 工数（計画） | 工数（実績） | 差異 | 主要課題 |
|-------------|------|------------|------------|------|---------|
| ECサイトA | 画面30、API50 | 800人日 | 920人日 | +15% | 決済API統合で遅延 |
| CRM実装B | 画面50、API80 | 1,200人日 | 1,150人日 | -4% | 要件が明確だった |
```

---

### 2. 段階的詳細化（Rolling Wave Planning）

**アプローチ**:
- 近い将来（2-3ヶ月）: 詳細見積もり（ボトムアップ）
- 中期（3-6ヶ月）: 中精度見積もり（パラメトリック）
- 長期（6ヶ月〜）: 概算見積もり（類推法）

**メリット**:
- 不確実性が高い遠い将来に時間をかけすぎない
- 情報が増えた時点で見積もりを更新
- リスクの早期発見

---

### 3. 複数手法による検証

**クロスチェック**:
1. ボトムアップで詳細見積もり
2. パラメトリック法で妥当性検証
3. 過去の類似プロジェクトと比較
4. 専門家レビュー

**差異が大きい場合**（±20%以上）:
- 前提条件を再確認
- 見落としているタスクがないか確認
- 調整係数が適切か検証

---

### 4. ステークホルダーレビュー

**レビュー観点**:
- **技術リード**: 技術的実現可能性、工数の妥当性
- **PM**: スケジュール、リソース配分の現実性
- **BA**: 要件の網羅性、業務適合性
- **営業/調達**: 市場価格との整合性、競合との比較

## 見積もり時の注意事項とリスク

### よくある見積もりミス

1. **楽観的バイアス**: 「うまくいく」前提で見積もる
   - 対策: 三点見積もりで悲観値も考慮

2. **アンカリング効果**: 最初に提示された数値に引きずられる
   - 対策: 独立して見積もり後に比較

3. **スコープクリープの未考慮**: 要件追加を織り込まない
   - 対策: コンティンジェンシー15%以上確保

4. **非機能要件の見落とし**: 性能、セキュリティ、可用性
   - 対策: 非機能要件チェックリスト活用

5. **統合・連携の過小評価**: 外部システム連携の複雑性
   - 対策: 統合テストに全体の15-20%割当

6. **プロジェクト管理工数の未計上**: PMの工数を忘れる
   - 対策: 全体の10-15%をPM工数として計上

7. **学習曲線の無視**: 新技術の学習時間を考慮しない
   - 対策: 習熟度係数1.3〜1.8を適用

8. **コードレビュー・リファクタリング**: 実装の20%を見込む
   - 対策: 実装工数の20%をレビュー・改善に割当

---

### リスク要因のチェックリスト

#### 技術リスク
- [ ] 新しい技術スタックを採用する
- [ ] パフォーマンス要件が厳しい（応答時間1秒以内等）
- [ ] 大量データ処理が必要（TB級）
- [ ] リアルタイム処理が必要
- [ ] 複雑なアルゴリズムが必要

#### 要件リスク
- [ ] 要件が曖昧または未確定
- [ ] ステークホルダーが多数（5名以上）
- [ ] ステークホルダー間で意見の相違あり
- [ ] スコープ変更が頻繁に発生する可能性

#### 統合リスク
- [ ] 5つ以上の外部システムと連携
- [ ] レガシーシステムとの統合
- [ ] API仕様が不明確
- [ ] データフォーマット変換が必要

#### チームリスク
- [ ] チームメンバーが初めて一緒に働く
- [ ] 技術スタックに不慣れ
- [ ] リモートワーク中心（コミュニケーション課題）
- [ ] チームメンバーの離脱リスク

#### プロジェクト管理リスク
- [ ] 納期が厳しい（通常の80%以下）
- [ ] 予算制約が厳しい
- [ ] 複数プロジェクトとリソース競合
- [ ] 意思決定者が多忙（承認遅延リスク）

**リスク対応**:
- 各リスクに対する対策を明確化
- リスクが3つ以上該当: コンティンジェンシー20%以上
- リスクが5つ以上該当: コンティンジェンシー30%以上、またはプロジェクト分割検討

## まとめ

**見積もり精度向上のポイント**:
1. プロジェクトフェーズに応じた手法選択
2. 過去実績データの蓄積と活用
3. 複数手法によるクロスチェック
4. 適切なコンティンジェンシー設定（10-25%）
5. リスクの早期特定と対策
6. ステークホルダーレビュー
7. 定期的な見積もり更新（要件明確化に伴い）

**見積もりは継続的なプロセス**:
- 初期見積もり（精度±30〜50%）
- 要件定義後（精度±15〜30%）
- 設計完了後（精度±10〜20%）
- 実装前（精度±5〜10%）

各フェーズで見積もりを見直し、精度を向上させることが重要です。
