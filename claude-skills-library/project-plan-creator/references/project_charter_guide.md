# プロジェクト憲章（Project Charter）作成ガイド

## Overview

プロジェクト憲章は、プロジェクトを正式に承認し、プロジェクトマネージャーに権限を与える文書です。プロジェクトの存在理由、目的、主要なステークホルダー、概算予算とスケジュール、成功基準を明確にします。

**PMBOK準拠**: PMBOK® Guide 第6版・第7版の統合管理（Integration Management）に基づく

---

## プロジェクト憲章の目的

1. **プロジェクトの正式な承認**: プロジェクトの実施を組織として承認
2. **PMの権限付与**: プロジェクトマネージャーにリソース活用の権限を与える
3. **ステークホルダーの合意**: 主要ステークホルダーの期待を文書化
4. **プロジェクトの境界設定**: スコープ、予算、スケジュールの概要を設定
5. **成功基準の明確化**: プロジェクトの成功を測る基準を定義

---

## プロジェクト憲章の構成要素

### 1. プロジェクト基本情報

| 項目 | 説明 | 記載内容 |
|------|------|---------|
| **プロジェクト名** | プロジェクトの正式名称 | 明確で識別可能な名称 |
| **プロジェクトコード** | プロジェクト識別子 | PRJ-YYYY-XXX 形式 |
| **発行日** | 憲章の発行日 | YYYY年MM月DD日 |
| **バージョン** | 文書バージョン | 1.0, 1.1等 |
| **承認日** | スポンサーによる承認日 | YYYY年MM月DD日 |

---

### 2. エグゼクティブサマリー

**目的**: プロジェクトの要点を1-2ページで要約

**含めるべき内容**:
- プロジェクトの背景（なぜこのプロジェクトが必要か）
- プロジェクトの目的（何を達成するか）
- プロジェクトの価値提案（組織にどんな価値をもたらすか）
- プロジェクトの期間と予算（概算）

**例**:
```markdown
## エグゼクティブサマリー

当社の受注処理システムは10年前に開発され、現在のビジネスニーズに対応できていません。
手作業による処理が多く、エラー率5%、平均処理時間30分/件と非効率です。

本プロジェクトは、受注処理システムを刷新し、自動化・効率化を実現します。
これにより、処理時間を5分/件に短縮（83%削減）、エラー率を0.5%に低減（90%削減）し、
年間18,620,000円のコスト削減と20,000,000円の売上増加を見込みます。

プロジェクト期間: 6ヶ月
プロジェクト予算: 40,000,000円
投資回収期間: 1.23年
```

### 3. プロジェクトの背景と目的

#### 3.1 プロジェクトの背景

**WHYを明確にする**:
- 現在の課題（As-Is）
- ビジネス上の問題
- 市場や顧客からの要求
- 競合状況
- 法規制やコンプライアンス要件

**例**:
```markdown
### プロジェクトの背景

#### 現在の課題
- 受注処理が手作業中心で、平均30分/件かかる
- エラー率5%により、顧客満足度が低下（CSAT 6.5/10）
- レガシーシステムで保守コストが高い（年間5,000,000円）

#### ビジネス上の問題
- 受注量の増加に対応できず、機会損失が発生
- 競合他社は自動化済みで、競争力が低下
- 処理遅延によりリードタイム24時間、顧客からのクレーム増加

#### 市場要求
- 顧客はリアルタイム処理を期待（目標: 1時間以内）
- モバイル対応の要求が増加
```

#### 3.2 プロジェクトの目的

**WHATを明確にする**:
- プロジェクトで達成すること
- ビジネス目標との整合性
- 定量的な目標（KPI）

**例**:
```markdown
### プロジェクトの目的

1. **業務効率化**: 受注処理時間を30分→5分に短縮（83%削減）
2. **品質向上**: エラー率を5%→0.5%に低減（90%削減）
3. **顧客満足度向上**: リードタイムを24時間→1時間に短縮、CSAT 8.5/10達成
4. **コスト削減**: 年間18,620,000円の運用コスト削減
5. **売上増加**: 処理能力向上により年間20,000,000円の売上増加
```

### 4. プロジェクトスコープ（概要）

#### 4.1 In Scope（スコープ内）

プロジェクトで実施する内容:

**例**:
```markdown
### スコープ内

**システム開発**:
- 受注処理自動化システムの開発
- 在庫管理システムとの連携
- 顧客ポータルの開発

**主要機能**:
- 自動受注処理
- AI によるデータ入力・検証
- リアルタイム在庫連携
- 顧客向けWebポータル

**データ移行**:
- 既存システムからのデータ移行（過去1年分）
```

#### 4.2 Out of Scope（スコープ外）

プロジェクトで実施しない内容:

**例**:
```markdown
### スコープ外

- 販売管理システムの刷新（別プロジェクトで実施予定）
- 過去1年以前のデータ移行
- 倉庫管理システムの更新
- 社内教育プログラムの全面刷新
```

### 8. プロジェクトの前提条件と制約

#### 8.1 前提条件（Assumptions）

プロジェクトが成功するための前提:

**例**:
```markdown
### 前提条件

1. **リソース**: 必要な開発リソースが確保できる
2. **ステークホルダー**: 主要ステークホルダーが週次会議に参加可能
3. **環境**: テスト環境が2週間以内に準備される
4. **データ**: 既存システムのデータが提供される
5. **承認**: 要件定義・設計の承認が2週間以内に完了する
```

#### 8.2 制約条件（Constraints）

プロジェクトを制限する条件:

**例**:
```markdown
### 制約条件

1. **スケジュール**: 2025年9月30日までに本番稼働必須
2. **予算**: 総予算40,000,000円を超過不可
3. **リソース**: 開発チーム最大8名
4. **技術**: 既存インフラとの互換性を維持
5. **法規制**: 個人情報保護法、電子帳簿保存法に準拠
```

## プロジェクト憲章作成のベストプラクティス

### 1. 簡潔に保つ

- 5-10ページ以内に収める
- 詳細はプロジェクト計画書に記載
- エグゼクティブサマリーで要点を伝える

### 2. ビジネス言語を使う

- 技術用語を避け、ビジネス価値を強調
- ROI、コスト削減、売上増加など定量的な価値を示す
- 経営層が理解できる言葉で記述

### 3. 明確な成功基準

- 測定可能な基準を設定
- SMART（Specific, Measurable, Achievable, Relevant, Time-bound）
- 成功の定義をステークホルダー間で合意

### 4. リスクを正直に記載

- 主要なリスクを隠さずに記載
- リスク対策を明示
- ステークホルダーの期待を現実的に設定

### 5. スポンサーの承認を得る

- エグゼクティブスポンサーの署名必須
- プロジェクトオーナーの承認
- 主要ステークホルダーの合意

## まとめ

プロジェクト憲章は、プロジェクトの「憲法」として機能します。

**重要なポイント**:
1. プロジェクトを正式に承認し、PMに権限を与える
2. プロジェクトの目的、スコープ、成功基準を明確にする
3. 主要ステークホルダーの期待を文書化する
4. 簡潔で、ビジネス価値を強調する
5. エグゼクティブスポンサーの承認を得る

プロジェクト憲章なしにプロジェクトを開始すると、目的が不明確、権限不足、ステークホルダーの期待のずれが発生します。必ずプロジェクト開始前に作成・承認を得ることが重要です。
