# ITIL 4 技術的マネジメントプラクティス ― 詳細解説

> **目的** : 本ドキュメントは ITIL 4 における 3 つの技術的マネジメントプラクティス（Deployment Management／Infrastructure & Platform Management／Software Development & Management）を深掘りし、社内共有用および教育コンテンツとして理解促進を図ることを目的とする。
>
> **対象読者** : IT/DevOps チーム、サービスマネジメント担当、プロジェクトマネージャ、システムエンジニア、インフラエンジニア、新人教育担当者 など

***

## 0. 技術的マネジメントプラクティスとは 

| カテゴリ      | プラクティス                               | 目的（ITIL 4 公式）                                                    |
| --------- | ------------------------------------ | ---------------------------------------------------------------- |
| 技術的マネジメント | Deployment Management                | *新規または変更済みサービスやリリース構成品を本番／テスト環境に移送・配置する*                         |
| 技術的マネジメント | Infrastructure & Platform Management | *サービスの提供・運用に必要な基盤（ハードウェア・ソフトウェア・ネットワーク・プラットフォーム）の計画・設計・展開・運用を行う* |
| 技術的マネジメント | Software Development & Management    | *ソフトウェア資産の計画・設計・構築・テスト・リリース・改善を通じてサービス価値を創出する*                   |

各プラクティスは **サービスバリューチェーン (SVC) の Obtain/Build → Design & Transition → Deliver & Support → Improve** を主に担い、ビジネス価値のフローを技術面から加速させる役割を持つ。

***

## 1. Deployment Management（デプロイメント管理）

### 1.1 目的

変更済みサービス・機能を **リスクを最小化しつつ迅速かつ確実に** 利用可能環境へ展開する。

### 1.2 適用範囲

- アプリケーション／ミドルウェア／インフラ設定のデプロイ

- テスト〜ステージング〜本番環境の同期

- フィーチャーフラグを用いた段階的リリース

### 1.3 主な活動

1. **デプロイ計画策定** : ビジネスカレンダー・メンテナンスウィンドウ・依存関係を考慮

2. **リリースパッケージ作成** : バージョン管理タグ＋アーティファクト署名

3. **自動デプロイオーケストレーション** : CI/CD パイプライン（例: GitHub Actions → ArgoCD）

4. **検証 & モニタリング** : 自動 Smoke Test / Synthetic Transaction

5. **ロールバック/ロールフォワード** : Chaos Engineering 演習を含む手順整備

### 1.4 役割と責任

| 役割           | 主な責任                   |
| ------------ | ---------------------- |
| デプロイメントマネージャ | 計画承認、リリースカレンダー統制、リスク評価 |
| DevOps エンジニア | パイプライン実装、自動テスト統合、観測性設定 |
| アプリケーションオーナー | 機能承認、リリース判定、KPI レビュー   |

### 1.5 KPI / メトリクス

- **Change Failure Rate (CFR)**

- **Mean Time To Recovery (MTTR)**

- **デプロイ頻度 (Deploys/Day)**

- **リリース成功率**

### 1.6 ベストプラクティス

- **Blue‑Green／Canary／Ring Deploy** で段階的に流量を増やす

- **Infrastructure‑as‑Code** で環境間ドリフトを解消

- **トランクベース開発** + **Feature Flag** で小さな変更を高頻度で投入

### 1.7 使用ツール例

| カテゴリ      | ツール                                  |
| --------- | ------------------------------------ |
| CI/CD     | GitHub Actions, GitLab CI, Jenkins X |
| デプロイ      | Argo CD, Spinnaker, Octopus Deploy   |
| フィーチャーフラグ | LaunchDarkly, Unleash                |
| テスト自動化    | Selenium, Playwright, K6             |

### 1.8 他プラクティスとの連携

- **リリース管理**（サービスマネジメント）とガバナンス統合

- **変更有効化 (Change Enablement)** と CAB 自動化

### 1.9 導入・改善のヒント

1. **現状のデプロイプロセスをマッピングし、ボトルネックを特定**

2. 手動ステップを 1 つずつ自動化 → モニタリング → 改善サイクル

3. DORA メトリクスを半年分収集し、品質・速度の相関を可視化

## 2. Infrastructure & Platform Management（インフラ & プラットフォーム管理）

### 2.1 目的

ビジネス要求に応じた **スケーラブル・レジリエント・コスト最適化された** プラットフォームを提供し、開発チームのバリューデリバリを加速する。

### 2.2 適用範囲

- IaaS/PaaS/SaaS 選定・設計・運用

- ネットワーク・セキュリティ・ストレージ基盤

- オブザーバビリティ／AIOps プラットフォーム

### 2.3 主な活動

1. **インフラアーキテクチャ設計** : Well‑Architected Framework 採用

2. **キャパシティ計画** : SLO‑based Capacity Model

3. **Infrastructure‑as‑Code (IaC) 実装** : Terraform + tflint + CI

4. **プラットフォーム運用** : Kubernetes Ops, Service Mesh, GitOps

5. **継続的改善** : Cost Optimization Report, Performance Review

### 2.4 役割と責任

| 役割                     | 主な責任                                     |
| ---------------------- | ---------------------------------------- |
| プラットフォームエンジニア          | 共通ランタイム／開発者ポータル提供                        |
| サイトリライアビリティエンジニア (SRE) | SLO 設定、Error Budget 管理、Incident Response |
| クラウドアーキテクト             | マルチクラウド戦略、セキュリティ・ガイドライン策定                |

### 2.5 KPI / メトリクス

- **SLO 達成率 / Error Budget Burn Rate**

- **インフラコスト／サービス単位**

- **プロビジョニングリードタイム**

### 2.6 ベストプラクティス

- **プラットフォームエンジニアリング** : 内製 PaaS + IDP (Internal Developer Platform)

- **オブザーバビリティ 3 本柱** : Metrics, Logs, Traces 統合ビュー

- **AIOps & 自動修復** : パターン学習 → アラート自動軽減

### 2.7 使用ツール例

| カテゴリ       | ツール                                |
| ---------- | ---------------------------------- |
| IaC        | Terraform, Pulumi, Ansible +       |
| コンテナ       | Kubernetes, OpenShift, EKS/GKE/AKS |
| GitOps     | ArgoCD, Flux CD                    |
| Monitoring | Prometheus, Grafana, Datadog       |
| AIOps      | New Relic AI, Moogsoft, Opsgenie   |

### 2.8 他プラクティスとの連携

- **情報セキュリティ管理** : IAM, Zero Trust, CSPM 連動

- **サプライヤ管理** : クラウドベンダ SLA、オンサイト保守契約

### 2.9 導入・改善のヒント

1. **Wardley Map** で自社の基盤をコモディティ化できる領域と差別化領域に分類

2. **FinOps** プラクティスと連携し、コストの見える化と改善ループを構築

3. SRE 手法で SLO → Error Budget → 運用投資判断 を標準プロセス化

## 3. Software Development & Management（ソフトウェア開発 & 管理）

### 3.1 目的

ソフトウェアによる **継続的な価値提供** を実現するために、アイデア発案から運用改善までを統合し高速に回す。

### 3.2 適用範囲

- 要件定義・ユーザストーリ作成

- コーディング・レビュー・セキュリティテスト

- リリース後のフィードバック・改善

### 3.3 主な活動

1. **価値ストリームマッピング (VSM)** : ビジネス → IT → Ops の時間可視化

2. **開発プロセス選定** : Scrum / Kanban / Scaled Agile Framework

3. **テストファースト／CI** : Unit → Integration → E2E, コード品質ゲート

4. **DevSecOps** : SAST / DAST / SBOM / Dependency Check

5. **リリースレトロスペクティブ** : KPI 分析 → 改善 Backlog 登録

### 3.4 役割と責任

| 役割                 | 主な責任                  |
| ------------------ | --------------------- |
| プロダクトオーナー (PO)     | 価値優先度付け、リリースゴール定義     |
| スクラムマスター / Dev リード | プロセスファシリテート、障害除去      |
| QA エンジニア           | 自動テスト設計、パイプライン品質ゲート管理 |

### 3.5 KPI / メトリクス

- **Lead Time for Changes**

- **コードカバレッジ／欠陥流出率**

- **顧客満足度 (CSAT) / NPS**

### 3.6 ベストプラクティス

- **シフトレフト** : 要件→テスト→セキュリティを前倒し

- **Inner Source** : 社内 OSS 文化で再利用性向上

- **ペアプログラミング & Mob Review** で知識共有

### 3.7 使用ツール例

| カテゴリ    | ツール                             |
| ------- | ------------------------------- |
| バージョン管理 | GitHub, GitLab, Bitbucket       |
| CI / 品質 | SonarQube, CodeQL, Dependabot   |
| テスト     | JUnit, pytest, Cypress, Postman |
| チケット管理  | Jira, Azure Boards              |

### 3.8 他プラクティスとの連携

- **継続的サービス改善 (CSI)** でレトロ改善項目を長期ロードマップに反映

- **リスク管理** : 技術的負債を可視化し解消計画を管理

### 3.9 導入・改善のヒント

1. **DORA メトリクスを四半期ベースでトレンド分析**

2. **価値ストリームを OKR と連動** し、組織目標と紐付け

3. **DevSecOps プレイブック** を作成し、開発者オンボーディングを高速化

***

## 4. 技術的マネジメントプラクティス共通の成功要因

1. **自動化とコード化 (Everything‑as‑Code)** で再現性と速度を両立

2. **観測可能性 (Observability) & フィードバックループ** : リアルタイムデータ → 改善バックログ化

3. **カルチャー & 人材育成** : Learning Organization + Blameless Postmortem

4. **ガバナンスとコンプライアンスの埋め込み** : Policy‑as‑Code / Guardrails

5. **継続的価値提供の測定** : ビジネス指標 (ARR, ユーザアクティブ率) と技術指標の相関を可視化

***

## 5. ケーススタディ (抜粋)

| 企業         | 背景                  | 施策                           | 効果                                    |
| ---------- | ------------------- | ---------------------------- | ------------------------------------- |
| FinTech A社 | リリースサイクル 4 週／高 MTTR | Canary Deploy + SRE 導入       | CFR ▼ 30%、デプロイ×5、MTTR ▼ 60%           |
| SaaS B社    | マルチクラウド負荷増          | IaC + GitOps + Cost Explorer | Provision LeadTime ▼ 70%、コスト最適化 △ 15% |

> **ポイント** : プラクティスを単体で導入するのではなく、**価値ストリーム全体** を見据えて統合・自動化することが成果につながる。

***

## 6. まとめ

- ITIL 4 の技術的マネジメントプラクティスは **“技術基盤をビジネス価値へ変換するエンジン”**。

- 成功には **自動化・観測性・継続的学習** の 3 本柱と、サービスマネジメント／一般的マネジメントプラクティスとの **相互連携** が不可欠。

- 本資料を活用し、現状プロセスを可視化 → ボトルネックを特定 → ベビーステップで改善 → メトリクスで効果検証 というサイクルを組織的に回してほしい。
